# 七步法项目管理核心经验

## 来源
AI自学平台MVP项目复盘（2026-03-21），七步法全流程实践总结。

## 五条核心经验

### E1 框架先行：先搭骨架再填充
- **场景**：用 Typer 搭建 CLI 时，先注册空的子命令组（kb/review/learn/sync），所有命令 --help 可用
- **效果**：后续每个工作包只需填充实现，入口架构从头到尾未变
- **适用**：所有 CLI/Web/API 项目。先定义路由和接口，再逐步实现具体逻辑

### E2 增量设计：同步记录水位线
- **场景**：飞书同步设计了 sync_record.json，记录每条笔记的同步时间
- **效果**：二次同步时 4 条笔记全部跳过（0 上传 / 4 跳过），节省 API 调用
- **适用**：任何涉及外部同步的功能。用时间戳/版本号记录同步位点，避免重复传输

### E3 Plan B 前置：策略阶段就定降级方案
- **场景**：Step 3 策略阶段为 WP4（AI推荐）规划了「无 API Key 时用本地规则引擎」的 Plan B
- **效果**：Step 6 确认无 API Key 时，零延迟切换方案，项目不受阻
- **适用**：所有有外部依赖的功能。在规划阶段就问：如果这个依赖不可用，我的 Plan B 是什么？

### E4 依赖要验证：资源阶段实测可用性
- **场景（教训）**：WP4 计划用 Claude API，但直到 Step 6 才确认没有 API Key
- **后果**：被迫降级，智能推荐能力受限
- **改进**：在资源保障阶段（Step 4），对每个外部依赖做实际调用测试，而非仅列入清单
- **适用**：第三方 API、云服务凭证、付费工具许可证等

### E5 测试即时：完成即测，不攒到最后
- **场景**：飞书脚本合并后立即运行 13 项集成测试，发现 load_dotenv(override=True) 问题并即时修复
- **效果**：Bug 在引入后几分钟内被发现和修复，而非累积到最后调试
- **适用**：所有开发工作。每完成一个模块/重构，立即运行相关测试

## 速查表

| 阶段 | 检查项 |
|------|--------|
| 规划 | WBS 拆到 ≤1 天粒度？验收标准明确？ |
| 策略 | 每个外部依赖有 Plan B？ |
| 资源 | 每个依赖实际测试过？ |
| 执行 | 骨架先行？完成即测？ |
| 监控 | 偏差 >10% 立即预警？ |
| 验收 | 逐项对照 SMART 目标？ |
