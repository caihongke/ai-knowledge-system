## 一、方案设计与评估

### WP0 项目基础设施

| 方案 | 描述 | 优点 | 缺点 | 成本 | 风险 |
|------|------|------|------|------|------|
| **A. Typer 框架 ✅ 已选** | 使用 Typer 搭建 CLI，pyproject.toml 管理依赖 | 类型提示自动生成帮助、代码简洁、学习曲线低 | 需安装第三方包 | 低 | 低 |
| B. argparse 标准库 | 使用 Python 内置 argparse | 零依赖 | 代码冗长、子命令管理繁琐、后期维护成本高 | 低 | 中 |

**决策理由**：Typer 代码量仅为 argparse 的 1/3，且自动生成 `--help`，适合快速迭代。

---

### WP1 知识库核心功能

| 方案 | 描述 | 优点 | 缺点 | 成本 | 风险 |
|------|------|------|------|------|------|
| **A. Markdown文件 + JSON索引 ✅ 已选** | 笔记存为 `.md` 文件，`index.json` 存储元数据（标题/标签/时间） | 人类可读、Git 友好、无需数据库、可直接编辑 | 大量笔记时检索较慢（>1000条） | 低 | 低 |
| B. SQLite 数据库 | 笔记内容和元数据全部存入 SQLite | 检索快、支持复杂查询 | 不可直接编辑、Git diff 不友好、学习成本高 | 中 | 中 |

**决策理由**：MVP 阶段笔记量小（<500），Markdown+JSON 足够；后期如需可迁移至 SQLite。

**全文检索实现策略**：
- 阶段一：Python 内置字符串匹配（`keyword in content`），零依赖
- 阶段二（如需）：引入 whoosh 库做倒排索引

---

### WP2 飞书云端同步

| 方案 | 描述 | 优点 | 缺点 | 成本 | 风险 |
|------|------|------|------|------|------|
| **A. 复用现有脚本封装 CLI ✅ 推荐** | 将 `feishu_final.py` 逻辑封装进 `cli/sync.py`，加增量同步 | 复用现有代码、开发量小 | 依赖飞书凭证可用 | 低 | 中 |
| B. 本地导出备份 | 导出为 JSON/HTML 到本地备份目录，不依赖飞书 | 零外部依赖、永远可用 | 无云端访问能力 | 低 | 低 |

**决策理由**：优先方案A，若飞书凭证修复失败则自动降级为方案B。

**增量同步策略**：在 `index.json` 中记录每条笔记的 `last_synced` 时间戳，仅上传 `updated_at > last_synced` 的笔记。

---

### WP3 间隔复习提醒

| 方案 | 描述 | 优点 | 缺点 | 成本 | 风险 |
|------|------|------|------|------|------|
| **A. 经典艾宾浩斯固定间隔 ✅ 推荐** | 固定间隔序列：1→2→4→7→15→30天 | 实现简单、算法成熟、易理解 | 不能根据掌握程度调整 | 低 | 低 |
| B. SM-2 自适应算法 | SuperMemo SM-2 算法，根据用户评分动态调整间隔 | 更科学、个性化 | 实现复杂、需用户每次评分 | 中 | 中 |

**决策理由**：MVP 用方案A快速上线；v2 可升级为 SM-2。

**数据结构**：
```json
{
  "note_id": "001",
  "created_at": "2026-03-24",
  "review_count": 2,
  "next_review": "2026-03-28",
  "interval_days": 4
}
```

### WP4 AI学习路径推荐

| 方案 | 描述 | 优点 | 缺点 | 成本 | 风险 |
|------|------|------|------|------|------|
| **A. Claude API 直接调用 ✅ 已选** | 通过 Anthropic SDK 调用 Claude，发送知识库摘要获取建议 | 效果最好、理解力强 | 有 API 调用成本（约 $0.01/次） | 中 | 中 |
| B. 本地 Ollama 模型 | 运行 llama3 等本地模型 | 零调用成本 | 效果差、需下载大模型、占资源 | 低 | 中 |

**决策理由**：Claude API 质量远超本地模型，MVP 阶段调用量低成本可控。

**成本控制策略**：
- 每次只发送笔记标题+标签摘要（非全文），控制 token 用量
- 缓存最近一次建议结果，24h 内重复查询不调 API
- 设置月度调用上限（如 100 次），超限提示用户

## 二、优先级排序（紧急-重要矩阵）

```
                    重要
                     ↑
         WP1 知识库  │  WP0 基础设施
         (重要+不急) │  (重要+紧急)
        ─────────────┼──────────────→ 紧急
         WP4 AI推荐  │  WP2 飞书同步
         (不急+不重) │  WP3 间隔复习
                     │  (紧急+不重)
```

| 优先级 | 工作包 | 象限 | 依据 |
|--------|--------|------|------|
| P0 | WP0 基础设施 | 紧急+重要 | 所有包的前置依赖，必须最先完成 |
| P1 | WP1 知识库 | 重要+不急 | 核心功能，是系统价值基础 |
| P2 | WP3 间隔复习 | 紧急+次重要 | 直击「遗忘」痛点，用户感知价值高 |
| P3 | WP2 飞书同步 | 次紧急 | 复用现有代码，2天即可完成 |
| P4 | WP4 AI推荐 | 不急 | 锦上添花，可降级到 v2 |

**执行顺序**：WP0 → WP1 → WP3 → WP2 → WP4

> 调整说明：将 WP3（间隔复习）提前到 WP2（飞书同步）之前，因为复习功能直击核心痛点，且不依赖飞书凭证。飞书凭证修复需等外部操作，放后面可减少阻塞风险。

## 三、风险预案（Top 3）

### 风险1：飞书凭证修复失败（概率：中 | 影响：中）

| 项目 | 内容 |
|------|------|
| **触发条件** | 飞书开放平台无法获取新 APP_SECRET，或应用审核未通过 |
| **影响范围** | WP2 飞书同步功能不可用 |
| **Plan A** | 登录飞书开放平台重新创建应用或刷新凭证 |
| **Plan B** | WP2 降级为「本地导出」：`sync export` 将知识库导出为 JSON+HTML 压缩包，飞书同步标记为 v2 |
| **决策点** | 03-28 开始 WP2 时，若 2h 内无法获取凭证则切换 Plan B |

### 风险2：WP1 知识库开发超时（概率：中 | 影响：高）

| 项目 | 内容 |
|------|------|
| **触发条件** | 全文检索或 Markdown 渲染比预期复杂，4天内无法完成 |
| **影响范围** | 整条关键路径延迟，挤压 WP3/WP4 时间 |
| **Plan A** | 检索先用简单字符串匹配，Markdown 渲染用 `rich` 库直接输出 |
| **Plan B** | 砍掉 T1.3（全文检索）和 T1.4（Markdown 渲染），MVP 只保留笔记 CRUD + 列表，检索推迟到 v2 |
| **决策点** | 03-26（第3天）若 T1.1/T1.2 未完成，立即启动 Plan B |

### 风险3：API 成本超出预算（概率：低 | 影响：中）

| 项目 | 内容 |
|------|------|
| **触发条件** | 开发调试阶段频繁调用 Claude API，或 prompt 过长导致 token 消耗大 |
| **影响范围** | WP4 开发成本不可控 |
| **Plan A** | 严格控制：只传摘要不传全文、缓存结果、设月度上限 |
| **Plan B** | 将 WP4 改为规则引擎：基于标签统计分析推荐（零 API 成本），AI 能力标记为 v2 |
| **决策点** | 开发过程中累计 API 费用 > $5 时切换 Plan B |
