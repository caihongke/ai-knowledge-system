## 四、复盘总结

### 成功经验（5 条）

**1. 框架先行，先跑通再迭代**
- 场景：WP0 搭建 Typer CLI 骨架时，先注册空的子命令组（kb/review/learn/sync），所有命令都能 `--help`
- 效果：后续每个 WP 只需填充实现，不用重构入口。CLI 架构从头到尾未变
- 可复用：任何 CLI 工具项目都应先搭骨架、注册路由，再逐步填充

**2. 增量同步设计避免重复工作**
- 场景：WP2 飞书同步设计了 `sync_record.json` 记录每条笔记的同步时间
- 效果：二次同步时 4 条笔记全部跳过（0 上传 / 4 跳过），节省 API 调用
- 可复用：任何涉及外部同步的功能，都应记录同步水位线

**3. 提前规划降级方案，遇阻不慌**
- 场景：Step 3 策略阶段就为 WP4 规划了「无 API Key 时用本地规则引擎」的 Plan B
- 效果：Step 6 发现无 API Key 时，直接切换方案，零延迟
- 可复用：对有外部依赖的功能，必须在策略阶段就定义降级方案

**4. 七步法流程保证了系统性**
- 场景：从诊断→蓝图→策略→资源→执行→监控→验收，每步都有产出物
- 效果：Step 6 监控时能精确量化偏差（52% 完成率、提前 6 天），决策有据可依
- 可复用：中等复杂度以上的项目都应有结构化的管理流程

**5. 测试先行，每完成一个模块立即验证**
- 场景：飞书脚本合并后立即运行 13 项集成测试；存储模块有 24 项断言
- 效果：合并脚本时发现 `load_dotenv(override=True)` 问题并即时修复
- 可复用：关键模块必须有自动化测试，重构后立即回归

### 失败教训（3 条）

**1. 外部依赖未提前验证**
- 场景：WP4 计划使用 Claude API，但直到 Step 6 才确认没有 API Key
- 影响：被迫降级为本地规则引擎，智能推荐能力受限
- 改进：在 Step 4（资源保障）阶段就应实际验证所有外部依赖的可用性，而非仅列清单

**2. 飞书凭证管理存在隐患**
- 场景：APP_SECRET 在项目初始化时就已失效，直到手动测试才发现
- 影响：`load_dotenv()` 缺少 `override=True` 导致环境变量缓存，调试耗时
- 改进：凭证类配置应有自动化健康检查（已有 health_monitor.py，但未覆盖凭证有效性）

**3. Step 5 执行计划未实时更新**
- 场景：每日待办看板始终停留在初始状态（42 任务全在 To Do 列）
- 影响：进度追踪依赖 Step 6 手动收集，而非看板自动反映
- 改进：应设计进度自动同步机制，或在每次开发结束后强制更新看板

### 意外事件

| 事件 | 影响 | 处理方式 |
|------|------|----------|
| 飞书 APP_SECRET 失效 | WP2 阻塞 | 用户手动重置，当日解决 |
| `load_dotenv` 不覆盖已有环境变量 | 测试假阳性（新 Secret 未生效） | 添加 `override=True` |
| Windows GBK 编码导致 subprocess 输出解码失败 | 集成测试崩溃 | 改用 bytes 模式 + `errors='replace'` |
| 两个飞书脚本功能重叠 | 维护成本翻倍 | 合并为统一 `feishu_upload.py` |
